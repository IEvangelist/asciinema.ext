# `scripts/` — demo recording pipeline

Two scripts here both produce the **same workflow** (GitHub Artifacts:
Explorer → paste a URL → pick an artifact → watch download progress →
open HTML preview in Simple Browser) but with very different tradeoffs.

## `demo/record-video.mjs` — Playwright-driven mockup recording

Renders an HTML mockup of VS Code under `scripts/demo/mockup.html`, walks it
through every state with scripted timing, and records the result to
`media/demo-explorer.webm` via Playwright's built-in `context.recordVideo`.

| | |
| --- | --- |
| **Output** | `media/demo-explorer.webm` (≈ 3 MB, ~34 s, 1280×800) |
| **Run** | `npm run record:demo` |
| **Prereqs** | `npm install` (installs the `playwright` devDep), then `npx playwright install chromium` once |
| **Deterministic?** | Yes — same output every run; no network or auth needed |
| **Visual fidelity** | Mockup, not real VS Code. Pixel-accurate to Dark+ theme but **does not exercise the actual extension code**. |

This is the video that ships in the README. Regenerate it any time you
change copy/colors/timing inside `scripts/demo/*`.

### Editing the mockup

- `scripts/demo/mockup.html` — single-page VS Code mockup. All scenes are
  pre-rendered in the DOM and toggled via `body[data-state="..."]`.
- `scripts/demo/playwright-report.html` — the fake Playwright HTML report
  loaded inside the Simple Browser iframe at the end of the demo.
- `scripts/demo/record-video.mjs` — the recorder. Spins up a tiny static
  server, installs helpers in the page (`window.__demo.setState`,
  `showChapter`, `showOverlay`), walks through each scene with explicit
  `waitForTimeout` pauses, and renames the autogenerated `.webm` to
  `media/demo-explorer.webm`.

## `record-demo.ps1` — Real VS Code recording (Windows)

Drives a **real** VS Code instance with the extension actually installed,
captures the window region via `ffmpeg gdigrab`, and produces a recording
of the live behavior against a real GitHub PR or workflow-run URL.

| | |
| --- | --- |
| **Output** | `media/demo-real.mp4` (size depends on duration / network) |
| **Run** | `pwsh -File scripts/record-demo.ps1 -PrUrl 'https://github.com/owner/repo/pull/123'` |
| **Prereqs** | Windows · `code` and `ffmpeg` on PATH · GitHub access to the target PR/run |
| **Deterministic?** | No — depends on network speed, artifact size, GitHub state, your foreground window |
| **Visual fidelity** | 100 % — it's real VS Code, real extension, real artifact pipeline |

### Important caveats

- **Foreground takeover.** SendKeys targets whatever window has focus.
  Don't touch your mouse or keyboard during the ~45 s the script runs, or
  keystrokes will end up in the wrong place.
- **Isolated profile.** The script uses `--user-data-dir` and
  `--extensions-dir` pointing at `$env:TEMP\asciinema-ext-demo-{profile,extensions}`,
  so it never touches your day-to-day VS Code settings or extensions list.
- **One-time auth.** The first run will prompt you to sign in with GitHub.
  The script holds for 60 s after launch specifically to let you complete
  that OAuth dance.
- **Timing knobs.** If your PR artifact is bigger/smaller than the default
  expectation, bump `-DownloadWait` (or any of the other `-*Wait` params —
  see `Get-Help scripts/record-demo.ps1 -Full`).
- **Dry run.** `-DryRun` prints the planned actions without launching
  anything.

### Why both scripts?

- The mockup recorder is reproducible, fast, and produces the same video
  every run — perfect for shipping in the README and the marketplace
  listing.
- The real VS Code recorder is the source of truth for **what the
  extension actually does today** — use it to validate that the mockup
  isn't drifting from reality, or to grab a fresh capture when the
  workflow changes.

If the mockup video and the real-VS-Code recording start to disagree,
that's a bug in `scripts/demo/mockup.html` or `scripts/demo/record-video.mjs`
— please open an issue or update the mockup to match.
